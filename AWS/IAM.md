1. INTRODUCTION TO IAM SERVICES
--------------------------------------------------------------------------------
AWS Identity and Access Management (IAM) is a web service that helps you 
securely control access to AWS resources. You use IAM to control who is 
authenticated (signed in) and authorized (has permissions) to use resources.

Key Concepts:
* Root Account: 
  - The single identity created when the AWS account is first made. 
  - Has complete, unrestricted access. 
  - RULE OF THUMB: Stop using it for daily tasks immediately after setup!
  
* Users: 
  - Persistent identities used by a person or service to interact with AWS.
  
* Groups: 
  - A collection of IAM users. 
  - You can specify permissions for a group, and any user in that group 
    automatically inherits them.
    
* Roles: 
  - An identity with permission policies that determine what the identity 
    can and cannot do in AWS. 
  - Unlike a user, a role does not have standard long-term credentials 
    (no password or access keys).
    
* Policies: 
  - JSON documents that define permissions. 
  - They can be attached to Users, Groups, or Roles.


--------------------------------------------------------------------------------
2. CREATING IAM USERS
--------------------------------------------------------------------------------
An IAM user represents the person or application that interacts with AWS.

Steps to Create a User:
[1] Log in to the AWS Management Console and open the IAM Console.
[2] In the navigation pane, choose "Users" and then click "Create user".
[3] Enter the User name (e.g., john_doe).
[4] Proceed to set up their access type and permissions.


--------------------------------------------------------------------------------
3. ASSIGNING CONSOLE AND PROGRAMMATIC ACCESS
--------------------------------------------------------------------------------
When creating or modifying a user, you must decide how they will interact 
with AWS.

* AWS Management Console Access: 
  - This is for human users who need to log in via a web browser.
  - You will set a password (autogenerated or custom).
  - You can require the user to create a new password at the next sign-in.
  - BEST PRACTICE: Always enable Multi-Factor Authentication (MFA).

* Programmatic Access: 
  - This is for applications, scripts, or developers using the AWS CLI or SDKs.
  - Instead of a password, this generates an Access Key ID and a Secret 
    Access Key.
  - WARNING: The Secret Access Key is only shown once at creation. If you 
    lose it, you must delete the old access key and create a new one.


--------------------------------------------------------------------------------
4. GROUP CREATION AND POLICY ATTACHMENT
--------------------------------------------------------------------------------
Instead of attaching permissions to individual users one by one, you should 
use Groups. This makes permission management much cleaner as your team grows.

Steps to Create a Group and Attach Policies:
[1] In the IAM Console, choose "User groups" and click "Create group".
[2] Enter a Group name (e.g., DevTeam or SysAdmins).
[3] Add users to the group by checking the boxes next to their names.
[4] Attach permissions policies. You can search for AWS Managed Policies 
    (like AmazonS3ReadOnlyAccess or AdministratorAccess) and check them.
[5] Click "Create group".

5. PRACTICAL DEMO: POLICY IMPLEMENTATION & RESTRICTION TESTS
--------------------------------------------------------------------------------
How policies actually work under the hood and how to test them.

The Anatomy of a JSON Policy (Example):
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:*",
      "Resource": "arn:aws:s3:::my-secure-bucket/*"
    }
  ]
}

Lab Demo: Testing Restrictions
[1] Create a User named "TestUser" with no permissions attached at all.
[2] Log in as "TestUser" in an incognito browser window.
[3] Navigate to the S3 Console. You will see API error messages stating 
    you do not have permission to list buckets.
[4] Go back to your Root/Admin account and create a group called "S3-List-Only".
[5] Attach the AWS-managed policy "AmazonS3ReadOnlyAccess" to this group.
[6] Add "TestUser" to this new group.
[7] Refresh the page on the "TestUser" browser. You will now be able to 
    see the S3 buckets and their contents, but you won't be able to delete 
    or upload files!

6. DIFFERENT TYPES OF ROLES & PRACTICAL DEMO
--------------------------------------------------------------------------------
Importance of Roles:
IAM Roles allow you to delegate access without sharing long-term security 
credentials. Instead of putting hardcoded access keys directly onto an EC2 
server, you assign a Role to the server. AWS handles the rotation of 
temporary credentials automatically!

Different Types of Roles:
* AWS Service Role: Used by AWS services (like EC2 or Lambda) to perform 
  actions on your behalf.
* Cross-Account Access Role: Allows users from one AWS account to access 
  resources in another AWS account without creating duplicate users.
* Identity Provider (IdP) / Federation: Allows users logged into your 
  corporate network or Google/Facebook to access AWS resources.

Practical Demo: EC2 Service Role
[1] Create the Role:
    - In IAM, go to "Roles" -> "Create role".
    - Select "AWS service" as the trusted entity, and choose "EC2".
    - Search for and attach the "AmazonS3ReadOnlyAccess" policy.
    - Name it "EC2-S3-Read-Role" and create it.
[2] Attach Role to EC2:
    - Go to the EC2 console and select a running instance.
    - Click "Actions" -> "Security" -> "Modify IAM role".
    - Select "EC2-S3-Read-Role" and save.
[3] The Test:
    - SSH into your EC2 instance.
    - Run the command: aws s3 ls
    - The instance will successfully list your buckets without you ever 
      having to run "aws configure" or input access keys!
